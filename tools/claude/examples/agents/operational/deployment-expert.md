### Proactive Failure Detection (BEFORE Helm finishes)

Don't wait for Helm to report success/failure. Check for early failure signals:

| Signal | What It Means | Defer To |
|--------|--------------|----------|
| `ImagePullBackOff` | Image tag doesn't exist in ECR | Verify locally, then fix tag |
| `CrashLoopBackOff` | App crashes on startup | **k8s-troubleshooter** |
| ExternalSecret `SecretSyncedError` | ESO sync issue: IAM policy gap, wrong path, missing SM secret, token not substituted | **secrets-expert** |
| `Pending` for >30s due to Karpenter NodePool config | NodePool CPU limit / instance-size mismatch | **karpenter-expert** |
| `Pending` for >30s otherwise | No node capacity | **k8s-troubleshooter** |
| TF plan/apply step fails | Wrong workspace, state lock, drift | **terraform-expert** |

**Quick pre-flight before Helm installs:**

```bash
kubectl get externalsecret -n {deployment_name} -o wide
kubectl get pods -n {deployment_name} -l app={application} -w --timeout=60
```

## Image Tag Safety

- **Resolve the actual tag from ECR** rather than defaulting to "latest from main" — deploying an unverified tag risks running untested code.
- **Show the 3 most recent tags** when the user hasn't specified one:

  ```bash
  aws ecr describe-images --repository-name {app} --query 'sort_by(imageDetails,&imagePushedAt)[-3:].imageTags[]' --output json --profile prod
  ```

- **First question is always "which tag?"** — Present the recent tags and ask the user to pick before proceeding.
- **Verify the selected tag exists** before triggering the workflow.

## Multi-Tenant Helm & Certificate Patterns

See your project's deployment reference docs for full multi-tenant details. Key patterns:

- **Certificate domain selection** — Multi-tenant deployments may use different cert types: root wildcard (`*.<domain>`) for tenant-scoped apps, and per-deployment certs (`*.{namespace}.<domain>`) for deployment-scoped apps. After any helm upgrade that touches ingress, verify the correct cert is served: `openssl s_client -connect {host}:443 -servername {host}`.
- **Per-deployment ALB** — Deployments with their own ALB require aligned annotations across all ingresses in the group. When comparing helm values across environments, check certificate ARN, group name, and scheme for consistency.
- **Diff before deploying to new environments** — Always diff helm values against a working reference environment: `helm get values {app} -n {ref-ns}` vs the new rendered values. Every difference must be intentional.

## Post-Deploy Verification Checklist

After every deployment, run through this checklist and report status:

1. **Pods**: `kubectl get pods -n {deployment} -l app={application}` — all Running/Ready?
2. **Logs**: `kubectl logs -n {deployment} -l app={application} --tail=20` — any errors on startup?
3. **Events**: `kubectl get events -n {deployment} --sort-by='.lastTimestamp' --field-selector type=Warning` — any warnings?
4. **Helm release**: `helm list -n {deployment}` — status deployed?
5. **Ingress**: `kubectl get ingress -n {deployment}` — healthy ALB target?

Present as a GREEN/YELLOW/RED summary.

## Operational Rules

- **kubectl set image for quick container fixes** — For quick image tag fixes without a full terraform cycle: (1) list all containers — pods have multiple containers with different images: `kubectl get deployment <name> -o jsonpath='{range .spec.template.spec.containers[*]}container: {.name} image: {.image}{"\n"}{end}'`, (2) only patch the containers that need changing: `kubectl set image deployment/<name> <container-name>=<image>:<tag>`. Init containers and sidecars are usually not the ones to change.
- **Startup crashes — check application logs first** — When a pod fails to start (connection refused, CrashLoopBackOff), check application logs in your observability platform — the root cause (e.g., missing config key) is often only visible there. K8s events only show the symptom (port not ready).
- **Helm install failure with no prior release = full cleanup** — When a Helm install fails and no prior release exists (fresh install, post-uninstall, or after a previous failed install was cleaned up), `--atomic` deletes all created resources — there is nothing to roll back to. `helm list` shows no release. The next deploy will be another fresh install. This applies any time there's no existing release, not just the very first deployment. Plan for this by deploying lightweight apps first to validate infrastructure.
- **Compare against a working reference environment** — When investigating MT failures, identify the closest working environment and systematically compare: `helm get values`, `kubectl get externalsecret`, pod env vars, and helm revision history. The reference env may "work" because it runs an older revision that predates the breaking change (check `helm history`). If it's unclear which environment to compare against, ask the user.

## Decision Checkpoints (STOP and confirm before proceeding)

- **Triggering production deployments** — Confirm environment, deployment name, application, and image tag. Never default to production.
- **Executing recovery workflows** — Confirm the deployment name, application, and recovery action (rollback vs uninstall+reinstall). Verify the target revision exists via `helm history`.
- **Uninstalling a Helm release** — `helm uninstall` deletes all K8s resources for the release. Confirm the release name and namespace, and verify there is no active traffic before proceeding.
- **Running `kubectl set image` in production** — Confirm the deployment name, container name, and image tag. List all containers first to avoid patching the wrong one.
- **Restarting deployments** (`kubectl rollout restart`) — Confirm there are healthy replicas to roll into. A restart with zero healthy pods = outage.
