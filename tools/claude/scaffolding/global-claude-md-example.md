# Example Global CLAUDE.md

This file shows what a production `~/.claude/CLAUDE.md` looks like. Copy to `~/.claude/CLAUDE.md` and customize.

---

```markdown
# Claude Code Global Rules

## General

1. Follow 12-Factor App design principles where applicable.
2. **Verify before asserting** — When unsure about CLI flags, API behavior, library features, or service capabilities, search online or check official docs. Do not guess.

## Multiple Repository Clones

The user has multiple clones: `~/projects/myapp` (primary), `myapp-2`, `myapp-3`.

- Always identify which clone at the start of work
- Be explicit about paths when referencing files across clones
- Don't assume which clone — ask if unclear

## Authentication

- AWS profiles: `dev`, `prod`. Default when unclear: **prod**
- If credentials are expired or wrong account, automatically run the appropriate SSO login command and continue. Do NOT stop and ask.

## Commit Message Policy

- Conventional commit format: `type(scope): description`
- No AI attribution — never include "Generated by AI", "Claude", "Co-Authored-By", or similar
- Concise — focus on what changed, not how

## Dangerous Commands

NEVER run destructive/irreversible commands without explicit user approval. This includes:
`delete`, `destroy`, `remove`, `rm`, `prune`, `force`, `hard`, `terminate`, or anything
that modifies state files/history, affects production resources, or cannot be easily undone.
STOP and ask first.

For cloud resource cleanup/deletion: create explicit KEEP/DELETE lists, get per-resource-type confirmation.

## Testing & Validation

- **NEVER dismiss unexpected diffs as "not our change" or "pre-existing".** Every diff is a finding.
- **When comparing against running infrastructure**, actually extract and compare real values. Don't just skim resource-level changes.
- **Report ALL findings**, not just the ones related to the current task.

## Correctness & Least Privilege

- **"Harmless" is not acceptable.** Something being non-breaking does not make it correct.
- **Every secret, env var, and permission must be justified.** Don't dismiss extra data as benign.
- **Accuracy over convenience.** Always verify what an app actually needs vs. what it receives.

## Agent Behavioral Constraints

- **Decision checkpoints** — Stop at decision points and present a plan before executing multi-step operations. Never chain destructive steps without approval between each.
- **Scope discipline** — Only modify files explicitly requested. Before touching adjacent files, state which file and why, and wait for approval.
- **Image tag discipline** — Never default to `main` or `latest` for container image tags without first resolving available tags from the registry. Always ask "which tag?" before proceeding.

## Documentation Output

- Investigation reports, setup guides, and ad-hoc documentation go to `~/docs/projects/` in an appropriately named directory.
- Only READMEs and code-adjacent docs belong in the repo.
```
