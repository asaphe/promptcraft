# Global CLAUDE.md Guide

How to design your personal `~/.claude/CLAUDE.md` — the cross-project behavioral layer that applies to every Claude Code session.

## What the Global CLAUDE.md Is

The global `~/.claude/CLAUDE.md` is your personal configuration that loads into every Claude Code conversation, regardless of which project you're in. It defines behaviors, preferences, and guardrails that should apply universally.

It sits above project-level CLAUDE.md in the configuration hierarchy:

```text
~/.claude/CLAUDE.md          (global — always loaded, all projects)
  └── .claude/CLAUDE.md      (project — loaded when in this repo)
       └── subdir/CLAUDE.md  (directory — loaded when in this directory)
```

## What Belongs Here vs Project CLAUDE.md

| Global (`~/.claude/CLAUDE.md`) | Project (`.claude/CLAUDE.md`) |
| ------------------------------ | ------------------------------ |
| Commit message format | Project architecture |
| Dangerous command guardrails | Service-specific build commands |
| Auth recovery behavior | Agent roster |
| Verification philosophy | Code standards for this repo |
| Multi-clone awareness | Multi-tenant patterns |
| Documentation output conventions | Subdirectory CLAUDE.md map |
| Tool/version discipline | Technology-specific rules |

**Rule of thumb:** If you'd want the behavior in a brand new project you haven't touched before, it belongs in global. If it only makes sense in the context of a specific codebase, it belongs in the project.

## Common Global Rule Categories

### Commit Message Policy

Control commit format and attribution across all projects:

```markdown
## Commit Message Policy

- Conventional commit format: `type(scope): description`
- No AI attribution — never include "Generated by AI", "Claude", "Co-Authored-By", or similar
- Concise — focus on what changed
```

### Dangerous Command Guardrails

Prevent destructive operations without explicit approval:

```markdown
## Dangerous Commands

NEVER run destructive/irreversible commands without explicit user approval. This includes:
`delete`, `destroy`, `remove`, `rm`, `prune`, `force`, `hard`, `terminate`, or anything
that modifies state files/history, affects production resources, or cannot be easily undone.
STOP and ask first.
```

### Auth Auto-Recovery

Don't stop and ask when credentials expire — fix it and continue:

```markdown
## Authentication

- Profiles: `dev`, `prod`, `mgmt`. Default when unclear: **prod**
- If credentials are expired or wrong account, automatically run the appropriate login command and continue. Do NOT stop and ask.
```

### Verification Culture

Prevent hallucinated answers about tools, APIs, and capabilities:

```markdown
## Verification

- **Verify before asserting** — When unsure about CLI flags, API behavior, library features, or service capabilities, search online or check official docs. Do not guess.
```

### Correctness and Least Privilege

"Harmless" doesn't mean correct:

```markdown
## Correctness & Least Privilege

- **"Harmless" is not acceptable.** Something being non-breaking does not make it correct. Unnecessary secrets, env vars, permissions, or resources are undesirable — they expand attack surface and violate least privilege.
- **Every secret, env var, and permission must be justified.** If an app doesn't need a value, it should not have it. Don't dismiss extra data as benign.
- **Accuracy over convenience.** Always verify what an app actually needs vs. what it receives.
```

### Scope Discipline

Only touch what's requested:

```markdown
## Agent Behavioral Constraints

- **Decision checkpoints** — Stop at decision points and present a plan before executing multi-step operations. Never chain destructive or state-changing steps without approval between each.
- **Scope discipline** — Only modify files explicitly requested. Before touching adjacent files, state which file and why, and wait for approval. Do not refactor, clean up, or "improve" code outside the request scope.
```

### Image Tag / Version Discipline

Never silently default to `latest` or `main`:

```markdown
- **Image tag discipline** — Never default to `main` or `latest` for container image tags without first resolving and showing the actual available tags from the registry. Always ask "which tag?" before proceeding.
```

### Multi-Clone Awareness

When you work with multiple copies of the same repo:

```markdown
## Multiple Repository Clones

The user has multiple clones: `~/projects/repo` (primary), `repo-2`, `repo-3`.

- Always identify which clone at the start of work
- Be explicit about paths when referencing files across clones
- Don't assume which clone — ask if unclear
```

### Testing and Validation Philosophy

Every diff is a finding:

```markdown
## Testing & Validation

- **NEVER dismiss unexpected diffs as "not our change" or "pre-existing".** Every diff in a plan or test output is a finding that must be investigated and reported.
- **When comparing against running infrastructure**, actually extract and compare real values. Don't just check exit codes and skim resource-level changes.
- **Report ALL findings**, not just the ones related to the current task.
```

### Documentation Output Conventions

Control where generated docs go:

```markdown
## Documentation Output

- **Investigation reports, setup guides, and ad-hoc documentation** go to `~/path/to/docs/` in an appropriately named directory.
- Only READMEs and code-adjacent docs belong in the repo.
```

## Anti-Patterns

### Putting project-specific content in global

```markdown
## Bad: This is project-specific
- Use yarn workspaces for the monorepo
- Data pipelines use partition keys for backfills
```

This only applies to one project. Put it in that project's `.claude/CLAUDE.md`.

### Duplicating project CLAUDE.md content

If both global and project CLAUDE.md say the same thing, the global one is redundant for that project and adds context window cost. Keep global lean — only rules that wouldn't exist in any project file.

### Making global CLAUDE.md too long

The global file loads into every conversation. Every line costs context tokens. Keep it under 100 lines. If a section grows beyond a few bullets, consider whether it's truly global or should be project-specific.

### Omitting the "why"

```markdown
## Bad
- Always use --profile prod
```

```markdown
## Good
- Default AWS profile when unclear: **prod** — dev changes are lower risk and more easily reversed.
```

The "why" helps the AI make correct decisions in edge cases not explicitly covered.
